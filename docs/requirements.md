# Animal Adventure L3 MVP Requirements

## MVP Goal

The L3 MVP goal is to deliver a locally deployable, browser-playable, progress-persistent, reconnect-capable minimum playable version.

Core loop:

Player opens browser -> enters name -> application creates or loads the player -> application owns a unique internal `player_id` -> player selects an MVP character only when creating a new player -> player enters the world map -> player can tour the full map while only Spawn-area gameplay is interactive -> player accepts quests from Hopper / Copper / Elisa -> player finds quest items near Spawn -> player returns to NPCs -> player receives coins/rewards -> player buys and uses Potion -> player automatically reaches L3 when requirements are met -> state persists -> player recovers after refresh, network disconnect, or backend process restart.

## In Scope

### Functional Requirements

- The game is playable directly in Chrome with no client installation.
- The game supports responsive layout for iPad and laptop screens.
- The player enters only `name` to create or load a player.
- `player_id` is generated by the backend and is never required as user input.
- Login is name-based only.
- Player name matching is case-insensitive and whitespace-normalized.
- The backend stores a unique normalized player name and returns the internal `player_id` after create/load.
- MVP enables 3 player character assets; later versions may expand to 9 characters.
- MVP character ids are `penguin`, `arctic_fox`, and `cat_snowman`.
- A new player chooses an MVP character during first creation.
- A returning player loads the existing character and does not reselect character in MVP.
- The map uses `assets/images/Items/game_map_full.png` as the source image, but the MVP client renders prepared 1024px map tiles from `assets/images/MapTiles/` using `config/map_tiles.json`.
- The map uses coordinate-driven interaction data, not a fine-grained tilemap. MVP hotspot data is intentionally empty because Portal scenery is non-interactive.
- Player movement, camera follow, and full-map bounds are supported.
- MVP allows full-map touring/observation, but only Spawn-area NPCs, Spawn-area quest items, shop, online player visibility, and preset chat are interactive.
- MVP Portal objects are non-interactive scenery: no hotspot, no Coming Soon modal, no scene transition, and no portal-enter log is required.
- Online multiplayer position sync is supported; other connected players are visible.
- Online player positions are kept in server memory during play. Movement must not write to SQLite on every update.
- Preset phrase chat is supported; free-text chat is out of scope.
- Three starter NPC quests are supported:
  - Hopper: find `item_blanket`; reward `$25` + `accessory_sleepy_hat`
  - Copper: find `item_bagpipe`; reward `$25`
  - Elisa: find `item_dance_shoes`; reward `$25`
- Quest accept, server-side timer start, countdown display, item spawn, touch pickup, quest turn-in, completion, failure, and cooldown are supported.
- Each NPC quest has at most one globally active quest instance at a time. While a player owns the active quest lock, other players cannot accept that NPC quest.
- Quest items are spawned only for the active lock owner. The quest lock is released when the quest completes or fails.
- The backend must run a quest expiry scanner every 30 seconds. It fails expired active quests, expires their item instances, starts failure cooldown, and releases NPC quest locks even if the player disconnects and never sends another quest message.
- Quest completion cooldown is 60 minutes per player.
- Quest failure cooldown is 30 minutes per player.
- Inventory, equipment, coins, Potion purchase, and Potion use are supported.
- L3 requirement: `unique_completed_quest_ids >= 2` and `used_potion_count >= 2`.
- Initial coins are `25`, enough to buy 2 MVP Potions at `$10` each. NPC quest rewards still grant coins, but L3 also requires at least 2 unique completed quests, so quests are required for progression and are not merely optional coin sources.
- SQLite persists player position, level, coins, quests, inventory, equipment, and map unlock state.
- WebSocket reconnect automatically restores server state.
- `state_sync` includes `server_time`, active quest `expires_at`, active/cooldown `cooldown_until` values, and active `world_items`.
- Bootstrap config is a hard dependency for gameplay. If `GET /api/v1/config/bootstrap` fails after its retry policy, the frontend must show a blocking error state with a manual retry option and must not enter the game world or open the gameplay WebSocket.
- WebSocket reconnect attempts continue for up to 120 seconds with exponential backoff. If reconnect does not succeed within that window, the frontend shows: "Server is temporarily unavailable. Please refresh the page."
- A single `player_id` may have only one active WebSocket session. When a new connection for the same `player_id` succeeds, the server sends an `error` message with code `duplicate_session` to the old connection and closes it.
- Map, NPC, quest, item, shop, character, preset phrase, asset, and progression data are configuration-driven.
- All static assets are served by Nginx.
- FastAPI serves only API and WebSocket traffic, not static files.
- Reaching L3 persists `playground` as an unlocked region and shows a level-up notification; Playground gameplay remains out of scope for MVP.

### Non-Functional Requirements

- Pages automatically adapt to different screen sizes.
- Initial load prioritizes map, player, NPCs, and core UI; music and non-critical assets load later.
- MVP uses the existing `assets/music/bgm-1.m4a` through `assets/music/bgm-4.m4a` as looping background music. Dedicated SFX are out of scope for MVP.
- Existing PNG assets should render clearly through Phaser scaling.
- If the FastAPI process crashes, a local watchdog script restarts it.
- If the whole machine reboots, recovery depends on an OS-level startup task documented in Ops.
- SQLite uses WAL, short transactions, periodic checkpointing, and cleanup policies.
- SQLite writes for gameplay mutations use short, atomic transactions; reward-granting operations must be idempotent.
- WiFi/network drops trigger automatic WebSocket reconnect and progress recovery.
- Logs must capture application lifecycle, functional paths, data flow, resource anomalies, errors, exceptions, and player behavior.
- Logs and persisted diagnostic data must not grow forever; rotation and cleanup are required.
- MVP local deployment uses Nginx as the single browser entrypoint.

## Out Of Scope

- L6/L9/L12/L15 gameplay.
- All 9 playable characters.
- Running, motorcycle riding, and Flash race.
- Pet system.
- Friend system.
- Player trading.
- Free-text chat.
- Portal minigames.
- Portal interaction or placeholder modal behavior.
- All non-MVP NPC quests.
- Collection book, hidden areas, and easter eggs.
- Fine-grained tilemap collision.
- Cloud deployment, formal account authentication, payments, and security audit.

## Assumptions And Defaults

- “Nigex” means Nginx.
- Default local browser entrypoint is `http://localhost:8080/`; port `80` is optional.
- Currency remains `$`.
- Login is name-based only.
- Backend owns generated `player_id`.
- WebSocket connection identity is the `/ws/{player_id}` path identity; message-body `player_id` must be ignored or verified against the connection identity.
- The frontend derives the WebSocket URL from `window.location.origin` by mapping `http://` to `ws://` and `https://` to `wss://`, unless `VITE_WS_URL` is explicitly configured.
- Chat uses preset phrase ids only.
- Map uses a full-size world coordinate system with tiled background rendering and coordinate-driven NPC/item/region data. MVP hotspot data remains empty.
- MVP world bounds are the full map rectangle. Gameplay interaction zones are limited to Spawn-area NPCs/items and global UI; non-Spawn landmarks are scenery until V2.
- WebSocket broadcast rate is 10Hz.
- The frontend must throttle outbound `player_move` messages to at most 20Hz.
- WebSocket broadcasts contain minimal player state needed for rendering, not full durable player snapshots.
- Position autosave is throttled to every 30 seconds, plus immediate save on quest accept, quest completion, quest failure, shop purchase, Potion use, level-up, and WebSocket disconnect.
- Quest timers use server UTC `expires_at`; client countdowns are display-only and must be restored from `state_sync` after reconnect.
- Login remains name-only for MVP. The login UI must clearly warn that the same normalized name loads the same local save; stronger identity such as PIN or formal accounts is V2+.
- Only Hopper, Copper, and Elisa quests are implemented.
- If `item_pillow` is missing, Hopper MVP only requires blanket.
- Exact MVP NPC coordinates may be adjusted in `config/npcs.json` without code changes.
- V2 begins after the L3 MVP loop. Reaching L3 persists `playground` unlock state and displays a level-up notification, but L3-after gameplay such as Playground interaction, Portal A, and NPC_9 is V2+.
